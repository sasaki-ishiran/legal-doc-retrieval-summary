# 法律文书智能检索与信息抽取系统论文初稿材料

本文档用于快速完成毕业论文初稿。正文写作时应结合学校模板调整标题层级、页眉页脚、摘要格式和参考文献格式。

## 题目建议

基于语义向量与混合检索的法律文书智能检索系统设计与实现

## 摘要写作要点

法律文书数量增长使得人工查找相似案例和理解裁判要点的成本不断提高。传统关键词检索具有较好的可解释性，但难以处理自然语言案情描述中的同义表达和语义相似问题；单一语义向量检索能够提升语义匹配能力，但在关键词命中和排序解释方面存在不足。针对上述问题，本文设计并实现了一套法律文书智能检索与信息抽取系统。

系统采用 Python 作为主要开发语言，使用 MySQL 存储法律文书数据，利用 SentenceTransformer 生成文书向量，并通过 FAISS 构建语义检索索引。在检索方法上，系统实现了 BM25 关键词检索、语义向量检索和混合检索三种模式，通过分数归一化与加权融合兼顾语义相关性和关键词可解释性。针对法律文书篇幅较长的问题，系统在智能分析模块中引入 RAG 分片思想，筛选关键证据片段后调用本地大模型生成案情摘要和关键要素；当大模型服务不可用时，系统使用规则方法兜底，保证演示和测试流程可运行。实验部分使用 Precision@K、Recall@K、MRR 和 nDCG@K 对不同检索模式进行比较。

实验与测试结果表明，系统能够完成法律文书导入、索引构建、文书检索、结果解释、摘要生成、要素抽取和检索评测等功能，为法律文书检索场景提供了一个具有工程完整性的原型方案。

关键词：法律文书；信息检索；BM25；语义向量；FAISS；混合检索

## 第 1 章 绪论

### 1.1 研究背景

随着司法公开和裁判文书数字化建设的推进，法律文书数据规模不断扩大。法律从业者、研究人员和普通用户在面对大量文书时，常常需要根据案情描述查找相似案例，并进一步理解案件事实、争议焦点、法院认定和裁判结果。传统人工查找方式耗时较长，单纯依靠标题或关键词检索也难以充分满足自然语言案情描述下的检索需求。

### 1.2 研究意义

法律文书检索系统不仅需要返回相关案例，还需要提供一定的解释信息，帮助用户理解检索结果为什么相关。BM25 等关键词检索方法具有较强可解释性，适合作为基础检索方法；语义向量检索能够从整体语义层面度量查询与文书的相似程度；混合检索则可以在二者之间取得平衡。将检索、摘要、关键要素抽取和实验评测整合在同一系统中，有助于提升毕业设计的工程完整性和论文实验支撑能力。

### 1.3 本文主要工作

本文主要完成以下工作：第一，设计法律文书智能检索系统的总体架构，明确数据层、检索层、智能分析层和展示层的职责；第二，实现 LeCaRD 数据导入、MySQL 存储和 FAISS 向量索引构建流程；第三，实现 BM25 关键词检索、语义向量检索和混合检索，并提供推荐理由和命中片段展示；第四，实现基于 RAG 分片的大模型摘要与关键要素抽取模块，并提供规则兜底机制；第五，设计检索评测模块，使用 Precision@K、Recall@K、MRR 和 nDCG@K 对不同检索方式进行比较。

## 第 2 章 相关技术

本章可介绍 Python、Gradio、MySQL、SentenceTransformer、FAISS、BM25、RAG 分片和检索评价指标。

BM25 是经典概率检索模型，综合考虑词项频率、逆文档频率和文档长度归一化，适合作为关键词检索基线。SentenceTransformer 可将查询和文书编码为稠密向量，FAISS 负责高效相似度搜索。RAG 分片用于处理长文书，将全文切分为多个片段并筛选与任务相关的证据内容，降低直接截断带来的信息遗漏风险。

## 第 3 章 系统需求分析

### 3.1 功能需求

系统应支持法律文书数据导入、文书检索、检索模式切换、Top K 参数调整、结果详情查看、推荐理由展示、命中片段展示、摘要生成、关键要素抽取和检索效果评测。

### 3.2 非功能需求

系统应具备较好的可运行性，在缺少真实数据或大模型服务时仍能通过演示模式和规则兜底完成基本展示；系统应具备一定可扩展性，便于后续增加重排序、筛选条件和更多评测样本；系统输出应保持可解释性，便于论文分析和答辩说明。

## 第 4 章 系统设计

系统整体分为配置层、数据层、索引层、检索层、智能分析层、评测层和展示层。配置层集中管理数据库连接、模型名称、索引路径和检索权重；数据层负责读取 LeCaRD JSON 文书并写入 MySQL；索引层负责生成文书向量并构建 FAISS 索引；检索层提供 BM25、语义检索和混合检索；智能分析层负责摘要和要素抽取；评测层负责计算检索指标；展示层通过 Gradio 提供交互界面。

数据库核心表为 `cases`，字段包括自增编号、案件编号、文书编号、案件名称、正文内容和判决结果。该设计能够支撑检索展示、索引构建和评测返回 ID 对比。

## 第 5 章 系统实现

`config.py` 实现统一配置，减少路径、数据库参数和模型参数散落在不同脚本中的问题。`data_import.py` 递归扫描数据集目录，将 JSON 文书清洗后写入 MySQL。`build_index.py` 从数据库读取文书，提取标题和关键正文片段，使用 SentenceTransformer 生成向量并写入 FAISS 索引。`retrieval.py` 实现 BM25 关键词检索、语义检索结果包装和混合检索融合。`llm_utils.py` 实现 RAG 分片、提示词构造、大模型调用和规则兜底。`evaluation.py` 实现 Precision@K、Recall@K、MRR 和 nDCG@K。`app.py` 构建 Gradio 页面并串联所有功能。

## 第 6 章 系统测试与实验分析

功能测试应覆盖系统启动、查询输入、检索模式切换、结果表展示、案例详情查看、AI 摘要与要素抽取、评测表生成等流程。检索实验应使用同一组人工标注查询，分别运行 BM25 关键词检索、语义检索和混合检索，并记录 Precision@5、Recall@5、MRR、nDCG@5。

## 第 7 章 总结与展望

本文设计并实现了法律文书智能检索与信息抽取系统，完成了数据导入、向量索引、三种检索模式、混合排序、摘要生成、关键要素抽取和检索评测等功能。系统通过 BM25 保留关键词可解释性，通过语义向量检索提升自然语言查询匹配能力，并通过混合检索综合二者优势。

后续可从以下方向改进：增加更多人工标注评测数据，提升实验结论可信度；引入交叉编码器重排序模型，提高排序精度；增加案由、法院、年份等筛选条件；接入法律条文知识库，提升法律依据解释能力；增加用户查询日志和数据统计页面，为系统优化提供依据。
